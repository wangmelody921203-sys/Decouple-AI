## 專案定位（v2）

本次方案不只交付可運行原型，而是把「客戶離場後仍能持續成長」納入交付核心。TechFlow 的三大痛點（成本、擴展性、幻覺）將用「共創 + 監測 + 回寫」三層機制處理，讓系統每次迭代都留下可重用能力，而非一次性輸出。

---

## 問題解法地圖（Problem-to-System Map）

### 1) Token 成本飆升
*   **對應角色:** 成本與規模優化師 + 連續訊號偵查官
*   **v2 作法:**
	*   建立 Prompt 分層策略與檢索觸發門檻
	*   每日輸出成本異常訊號（高峰時段/高耗上下文）
	*   將有效節流策略回寫到 shared-memory
*   **驗收指標:** 同品質前提下，成本降幅達標且波動可追蹤

### 2) 擴展性風險
*   **對應角色:** 成本與規模優化師
*   **v2 作法:**
	*   把關鍵流程拆成可替換模組（路由、知識層、回覆層）
	*   明確標註模型切換策略與退化方案
*   **驗收指標:** 併發提升時維持可運行，且不需整體重構

### 3) 幻覺與錯誤承諾
*   **對應角色:** 共創引導教練 + 連續訊號偵查官
*   **v2 作法:**
	*   建立「高風險回答」檢驗清單
	*   對錯誤案例建立 Failure-to-Fix Log
	*   每輪交付前執行 /pulse-check
*   **驗收指標:** 關鍵場景誤答率下降，且每次錯誤有可追溯修正紀錄

## 新版 COO 流程（含微迭代回圈）

### 階段 1：共創門禁與目標對齊
確認客戶參與強度與決策權責，定義本輪可驗收目標。

### 階段 2：小步交付與即時修正（微迭代回圈）
每一輪都做：
1. 交付最小可測版本
2. 收集客戶阻塞點
3. 立刻修正提示與流程
4. 回寫記憶（/memory-sync）

### 階段 3：交接賦能與自走檢核
交付原型、手冊、架構圖外，新增 7 日行動清單與失敗修正紀錄，確保客戶離場後可自走。

### 階段 4：連續監測與預警服務
由連續訊號偵查官提供 24/7 監測，週期性輸出風險提醒與下一步建議。

## Specialist 協作順序（v2）

1. 營運長 COO：定義本輪目標與驗收閥值
2. 共創引導教練：轉譯業務需求與評判規則
3. 極速 MVP 驗證官：產出可運行最小版本
4. 成本與規模優化師：做架構與成本雙優化
5. 連續訊號偵查官（新增）：持續監測、預警、記憶回寫

## 預算與合作模式（v2）

維持「基礎門票費 + 節省成本分潤」邏輯，但在分潤判定中加入品質門檻：
* 成本下降若伴隨關鍵品質下降，不計入有效分潤。
* 僅對「成本與品質雙達標」的優化成果啟用分潤。

---

## v1 vs v2 主要差異（供 Section 5 對照）

1. **服務形態差異：** v1 以工作坊交付為主；v2 新增連續監測與預警服務。
2. **角色結構差異：** v1 為 3 specialist；v2 新增原生型角色「連續訊號偵查官」。
3. **流程邏輯差異：** v1 偏階段式推進；v2 加入每階段小步驗證與修正回圈。
4. **交付深度差異：** v2 多出 7 日行動清單與 Failure-to-Fix Log，強化客戶離場後可操作性。
5. **商業規則差異：** v2 在分潤計算加入品質閥值，避免只追成本。
6. **記憶機制差異：** v2 明確加入 /memory-sync，將每輪有效經驗沉澱成下輪能力。
