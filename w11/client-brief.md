# Week 11 Client Brief

## 五格框架

### 1) 我的客戶是誰
- 客戶名稱：TechFlow Innovations（林先生）
- 角色與背景：5 人電商客服自動化新創的創辦人，懂基礎開發但缺架構與評判機制
- 當前階段：已有 GPT 原型，但成本高、品質不穩、擴展風險高

### 2) 他表面上說想要什麼
- 明確要求：2 週內交付可運行客服 AI，降低錯誤回覆與 token 成本
- 交付期待：可運行原型、輸出評判手冊、成本基線與優化對照
- 時程／預算邊界：2 週驗證，採基礎費用加節省成本分潤

### 3) 他真正想要的是什麼（最關鍵）
- Q1：他說出口的需求背後，真正擔心的是什麼？
	擔心客服量一上來就失控，造成品牌信任下降與退貨率上升。
- Q2：他沒說、但其實有的需求是什麼？
	他真正要的是團隊可獨立判斷 AI 回覆品質的能力，而不是只靠外包修 bug。
- Q3：如果我只交文件、不改決策能力，會失敗在哪裡？
	團隊仍無法持續維運，下一次需求變動就回到高成本救火模式。

### 4) 我們公司這次要做的核心承諾
- 這次最小可驗證成果（MVP）：
	交付可運行客服決策原型（含回覆前檢查、風險標記與人工覆核入口）。
- 可追蹤指標（成本、品質、可擴展性）：
	1) 平均單輪 token 成本下降 25% 以上
	2) 高風險錯誤回覆率下降 40% 以上
	3) 併發提升到 3 倍時不需重構核心流程
- 下輪可重用流程：
	每次交付前跑 /pulse-check，結束後跑 /memory-sync 回寫有效策略與錯誤修正。

### 5) Deliverable 形式（擇 3-5 項）
- [x] 可運行原型
- [x] 架構圖（含模型解耦）
- [x] 成本基線與優化對照
- [x] AI 輸出評判手冊
- [x] 7 日行動清單
- [x] Failure-to-Fix Log
